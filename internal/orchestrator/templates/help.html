Operator Workflow

  1. Start on /ui and find workers marked degraded, dormant, or probing.
  2. Open the failing worker page and record the failure stage, signature, and workload shape.
  3. If the worker is dormant, treat that as a known-bad paused worker, not a recovered one. Dormant means compute was intentionally stopped to avoid paying for repeated failures.
  4. Check the worker telemetry badge before trusting runtime fields. legacy telemetry means those fields are advisory until the fleet image is refreshed.
  5. Use Grafana to see whether the failure is isolated or clustered by profile or replay dataset.
  6. Copy the AI prompt or incident JSON, then run the listed Fly triage commands.

Control Plane vs Grafana

  • Use the control plane for worker identity, incident context, machine metadata, and next commands.
  • Use Grafana for fleet posture, workload shape comparison, sync age drift, restart counters, and failure clustering.
  • If the same signature appears across multiple profiles at once, suspect a shared subsystem first.

Post-Merge Verification

  1. Open /api/deployments/latest or the Latest Rollout card on /ui.
  2. Confirm the new SHA is recorded and updated_workers reaches the fleet size.
  3. If any workers are probing, wait for the next verification cycle and check whether they return to running.
  4. If the rollout shows beyond 45m grace, treat it as a stuck rollout and open the affected workers before merging more changes.
  5. If workers remain degraded or go back to dormant, open those worker pages and inspect the latest failure before retrying.

How To Use AI

Each worker page includes multiple prompt modes. Pick the prompt that matches your current question.

Ask the model to rank hypotheses, cite evidence from the incident bundle, and give exact next commands before proposing code changes.

When a failed worker has a failure_debug_snapshot, use that first. It includes verifier substep timing, restore-plan metadata, process and FD evidence, socket state, disk and cgroup snapshots, child exit evidence, and the worker tools available for live follow-up.

Failure Families

  • sync: Litestream control socket missing or timing out. Check /data/litestream.sock, the Litestream process, and restart behavior.
  • restore: replica object fetch, restore plan, or missing LTX trouble. Check restore logs and object-store behavior.
  • integrity_check: restore completed but validation failed. Focus on restore correctness and integrity output.

What Good Looks Like

  • You can explain the failing workload shape without SSHing anywhere.
  • You can tell whether the issue is clustered or isolated before opening logs.
  • You can hand the AI prompt bundle to another engineer and they can start immediately.
